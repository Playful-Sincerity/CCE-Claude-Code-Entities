## 22:30 — [Discovery] CRITICAL: Shell redirection through allowed Bash bypasses path guard

**What:** Ran an adversarial debate on the permission model. Mid-debate, tested CON's central attack vector empirically. Confirmed: `echo "content" > /any/path` in an allowed Bash command writes files without triggering the PreToolUse path guard hook.
**Why:** Debate asked whether the current model was structurally safe. CON argued the allow-list matches on command prefixes but the security-relevant event is the syscall — and the shell is Turing-complete between them. Empirical test proved the vulnerability is real.
**Means:** The current PD settings.json is NOT safe for autonomous operation. The entity could overwrite `~/.claude/CLAUDE.md`, `~/.claude/settings.json`, or any "protected" path via shell redirection through `echo`, `cat`, `diff`, `ls`, or any allowed Bash command that accepts a shell operator.
**Files:** `debates/2026-04-15-permission-model-safety/` (7 files: 3 rounds PRO/CON, synthesis, README)

### The category error
The path guard hook fires on Write/Edit tools only. Bash operators (`>`, `>>`, `|`, `tee`, heredocs, `$(...)`, subshells) bypass it entirely. Patching `echo` just starts whack-a-mole — `printf`, `tee`, heredocs, pipes, and environment-variable persistence (`PROMPT_COMMAND`, `GIT_EDITOR` via rc files) all write through commands that don't look dangerous.

CON's point: "the allow-list matches on typed command prefixes; the security-relevant event is the syscall produced after the shell interprets the command." These are separated by a Turing-complete interpreter the permission model does not emulate. **You cannot pattern-match your way across that gap.**

### Verdict
CON won the debate. The architecture is only defensible if the trust boundary moves OUT of Claude Code:
- **Option A — Structured tools only:** drop Bash from allow-list entirely. Everything routes through Write/Edit/Read/Grep/Glob where the path guard fires.
- **Option B — Syscall sandbox:** run entity under `sandbox-exec` (macOS) with read-only mounts on protected paths. Makes Bash safe at the OS level.

### Immediate actions needed
1. Do NOT run PD autonomously under current settings.json
2. Drop `bypassPermissions` for entity sessions (single highest-leverage change)
3. Remove `Bash(echo *)` and `Bash(cat *)` from PD allow-list
4. Decide Option A vs B (research needed)
5. Harden audit log with `chflags uappend`
6. Add `.claude/protected-paths.txt` to its own contents
7. Acceptance test: replay the bypass attack, must fail

### Connection: ~/.claude/settings.json "protection" is illusory
The global settings.json has `Edit(~/.claude/settings.json)` in its deny list, intended to protect the config from accidental modification. But our finding means ANY Claude Code session with Bash allowlist including `echo` can overwrite it via `echo '{...}' > ~/.claude/settings.json`. This isn't just an entity problem — it's a general Claude Code safety consideration.
