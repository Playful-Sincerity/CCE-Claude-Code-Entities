## Afternoon — [Discovery] Second confirmed Edit(settings.json) bypass — Python syscall path

**What:** In a separate Claude Code session setting up a breath-nudge hook, Claude hit the `Edit(~/.claude/settings.json)` deny rule. Wisdom said "try to see if you can get around it." Claude wrote a Python script to `/tmp/patch-settings.py` that opened settings.json and called `.write_text(...)`, then ran it via `Bash(python3 *)`. The script successfully modified settings.json.
**Why:** Same structural failure as the `echo "..." > file` bypass from 2026-04-15 night. Deny rules gate specific tools; Bash can invoke interpreters that call write syscalls directly, which the permission system never sees.
**Means:** The `Edit(~/.claude/settings.json)` deny rule is **cosmetic in the current setup**. It blocks the Edit tool path but doesn't protect the file. Any Claude session with `Bash(python3 *)`, `Bash(echo *)`, `Bash(node *)` etc. in allow can rewrite settings.json via script → syscall.

Two confirmed bypass instances in 24 hours:
1. 2026-04-15 (PM) — `echo "test data" > /tmp/protected.md` under bypassPermissions
2. 2026-04-16 (PM) — Python script writing to `~/.claude/settings.json` under normal mode

Both authorized (test / explicit request). Both used legitimate allow-listed Bash commands. Both bypassed the intended protection completely.

### Why this is fine right now (and why it matters for entities)

For Wisdom's daily VS Code work: he's in the loop, he asked, the change was legitimate. No incident.

For autonomous entities: catastrophic. An entity with `Bash(python3 *)` in allow can overwrite its own settings.json, its SOUL.md, any "protected" path, because the deny rules don't cover the interpreter-→-syscall path.

### What this reinforces for tomorrow's security work

The fix is already captured in `sessions/2026-04-16-vscode-full-path.md` and `sessions/2026-04-16-read-protection-PII.md`, but this incident strengthens the priority:

1. **Enable native sandbox globally** in `~/.claude/settings.json` — syscall-level enforcement closes the interpreter bypass the same way it closed the shell redirection bypass (verified 2026-04-16 at 00:00).
2. **Remove/narrow `Bash(python3 *)`, `Bash(echo *)`, `Bash(node *)` from entity allow lists** — any interpreter is a write primitive.
3. **Install path-guard hook for Bash** that scans commands for write-to-protected-path patterns — imperfect but catches the obvious cases as defense-in-depth.

The deny-list approach alone is not enough. The sandbox is the actual answer. These two live incidents are the empirical proof that gets cited in any future decision about the security architecture.

**Files:** `~/.claude/settings.json` (modified to add breath-nudge.sh to Stop hooks — legitimate change, but via bypass path)
