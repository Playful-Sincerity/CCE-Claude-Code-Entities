## 13:35 — [Discovery]

**What:** Ran live verification of Task 01 against a throwaway entity (`/tmp/test-entities/test-harden-02`). 5 of 6 deny checks passed immediately. `echo hello` slipped through — the supposedly-removed allow entry still resolved.
**Why:** Claude Code **unions allow lists** across user-global + project settings, but **deny wins over allow** across all levels. Our user-global `~/.claude/settings.json` allows `Bash(echo *)`, `Bash(cat *)`, `Bash(diff *)`. Removing them from the entity's project-level allow did nothing — user-global re-granted them.
**Means:** The task brief's "Remove from allow" recipe is insufficient whenever user-global also allows the pattern. The only cross-level-safe block is an explicit deny. Added `Bash(echo *)`, `Bash(cat *)`, `Bash(diff *)` to the entity template's deny list and to PD's migrated settings. Re-verified against a fresh spawn — all 6 spec cases now pass (2 positive succeed, 4 negative blocked).
**Branch:** `main` (working-tree edits)
**Files:** `scripts/spawn-entity.sh`, `psdc/.claude/settings.json`

General lesson for Phase 1 going forward: when hardening an entity, audit user-global allow for anything dangerous-to-entities and mirror it in the entity's deny list. Allow-list removal is cosmetic across inheritance; only deny-list addition actually blocks.
